# Avoiding & resolving merge conflicts

This repo is active — `main` moves fast (dependabot, feature PRs). A few habits keep merges clean.
Most conflicts we saw came from three things: **stale branches**, **committed generated files**, and
**a handful of hot shared files** touched by nearly every PR.

## Prevent

- **Keep branches short-lived and current.** Before opening a PR, and again whenever `main` moves,
  sync your branch:

  ```sh
  git fetch origin main
  git merge origin/main      # or: git rebase origin/main
  ```

  A branch that sits for weeks falls hundreds of commits behind and conflicts everywhere. Sync early,
  sync often — a small merge every day beats one giant merge at the end.

- **Never commit build output.** `public/sitemap.xml` and `public/robots.txt` are regenerated by
  `build:sitemap` on every `npm run build` and are git-ignored. Don't `git add -f` them.

## Resolve

- **`package-lock.json` / `functions/package-lock.json`** — never hand-edit. Resolve `package.json`
  first (keep both sides' dependency additions), then regenerate the lockfile:

  ```sh
  git checkout --theirs package-lock.json   # or --ours; contents get replaced anyway
  rm package-lock.json && npm install       # regenerates a consistent lockfile
  git add package.json package-lock.json
  ```

  (Run inside `functions/` too if that lockfile conflicted.)

- **`src/i18n/en.json` / `src/i18n/ar.json`** — these are append-heavy, so two branches adding copy
  almost always collide. The changes are **additive**: keep **both** sides' new keys. Afterward
  `npm test` enforces en/ar key parity, so a dropped key fails CI rather than shipping.

- **Everything else** — read both versions and merge intent by hand. Then run `npm run verify`
  (and the `functions/` gate — `npm run lint && npm test && npm run build` — if you touched
  `functions/`) before pushing.

## Why no auto-merge driver?

`merge=union` on the i18n JSON would concatenate both sides and produce **invalid JSON** (duplicate
keys, broken commas). Lockfiles must be regenerated, not textually merged. So there is deliberately
no magic `.gitattributes` merge driver for these — the steps above are the safe path. `.gitattributes`
only normalizes line endings (LF everywhere) and marks generated files, which removes the _other_
class of noise conflicts.
